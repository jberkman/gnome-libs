Introducción

Gnome se esfuerza en proveer al usuario con el máximo nivel de configurabilidad, y al programador con el máximo nivel de comodidad. Afortunadamente, estas dos metas tienden a ir de la mano: Pocas decisiones para el programador significa muchas decisiones para el usuario.

Esta sección describe las facilidades de Gnome para comunicar mensajes simples o preguntas al usuario, y recuperar una respuesta. Los widgets principales involucrados son GnomeDialog y GnomeAppBar (una barra de estado, pero este nombre se desechó). Gnome también provee de funciones útiles las cuáles dan al usuario una gran alternativa como un efecto lateral. Por ejemplo, si llamas a la función gnome_app_message() en lugar de crear un diálogo directamente, el usuario puede solitar que los mensajes le aparezca en la barra de aplicaciones.

Alguna funcionalidad se ha perdido, y alguno de los API no está totalmente optimizado. Esperemos que esto se solucionará proximamente. En particular:

  • GnomeAppBar no funciona en modo interactivo, asi que todas las preguntas se deben hacer via díalogo. Toda la funcionalidad del Emacs-minibuffer-style se tiene que añadir.

  • GnomeAppBar deberia usar GnomePreferencesType para sus parametros has_progress , y no usar una barra de progreso si el usuario quiere díalogos de progreso en su lugar.

  • Llamadas a funciones con punteros no estándard son un pobre substituto para las señales Gtk y molesto con las ligaduras distintos de C. Una posible solución es crear un GtkObject para representar la interacción con el usuario, completado con señales. GnomeQuestion, por ejemplo, con una señal de "answered" y "cancelled". Más complejo, pero más flexible.

  • It is all sort of clunky feeling. ¿Quizás las funciones utiles no son elegantes inherentemente?.

Parches y sugerencias son siempre bienvenidos. Por favor piensa mientras lees esto como el API se podría mejorar y usa las facilidades, entonces envianos tus ideas.
